首页待办与日历功能审查总结
========================================
项目：ai-workbench
审查范围：CalendarWorkspace、DayPreviewPanel、CalendarMonth、DashboardTopBar、todo.service
生成日期：2026-06-10

一、关键问题（建议优先处理）
----------------------------------------

【1】切换月份后，日历数据不会跟着变
- UI 支持左右切换月份/周，但 loadTodos() 只调用一次 GET /smart-todo/month-list，且没有传 year/month 参数。
- changePeriod() 仅更新前端 currentMonth，不会重新请求数据。
- 后果：用户切到非当前月时，月历可能显示为空，误以为该月没有待办。
- 建议：与后台确认 month-list 是否支持按月份查询；若支持，切换月份时带参数并 refreshTodos()。

【2】左侧「本月任务」统计与日历当前月不一致
- monthEvents 基于 todayDate 所在自然月计算，而非日历正在浏览的 currentMonth。
- 后果：用户在日历切到 7 月，左侧仍显示 6 月统计，数据口径不一致。
- 相关代码：CalendarWorkspace.vue 中 currentMonthDates 计算逻辑。

✅【3】他人派发的待办：点击后可能「没反应」
- openEventDetail() 对未完成项走 openEditForm()，而 editable 仅创建者可编辑。
- 后果：非创建者点击卡片/条目可能无法打开任何界面。
- 建议：对 !editable && status !== 'done' 也进入只读「查看」模式（类似已完成待办）。

✅【4】AI 解析失败时无用户提示
- DayPreviewPanel.vue 中 parseTodoText() 只有 try/finally，没有 catch。
- 后果：接口失败时用户看不到 toast，只会停止 loading。
- 建议：补充 catch，展示错误提示。

【5】待办状态展示过于粗糙
- 后台状态：0 待处理、3 已接受、6 已完成、9 已拒绝、99 已删除。
- 前端仅映射为 todo | done 两种视觉状态。
- backendStatus 已保存但未充分用于 UI，「待接受」「已拒绝」在日历侧栏无法区分。

四、数据与权限注意点
----------------------------------------

1. 用户 ID 对齐
   - 权限判断依赖 getInfo 返回的 userName 与后台 assigneeId/creatorId/currentHandlerId 一致。
   - 不一致会导致无法完成、无法编辑、通知列表异常。

2. 撤销完成
   - 当前实现：先 GET /smart-todo/{id} 取详情，再 PUT /smart-todo 并传 status=3。
   - 需与后台确认是否要求传全量字段，以及 status=3 是否为约定的撤销后状态。

3. 完成接口
   - 仍通过 userId 请求头传当前用户，需确认后台是否仍必需。

4. 列表字段完整性
   - month-list / pending-list 若缺少 creatorId、currentHandlerId、remark 等字段，
     权限和展示会偏保守（例如误隐藏编辑/完成按钮）。

5. 已完成待办
   - 已实现只读查看模式，不可编辑；可通过「撤销」恢复为未完成。

五、体验与交互可优化
----------------------------------------

- 通知面板：接受/拒绝/转发后会刷新日历，但日历操作不会反向刷新通知角标。
- 负责人选择：通知转发已改内联列表；侧栏 Select 在复杂布局下需关注定位问题。
- 跨天待办：月视图有「跨天：每日/横条」切换，需确认与后台 timeType=2 展示一致。
- 周视图：时间轴较完整，但数据源仍受「仅当前月列表」限制，跨月周可能缺数据。
- 节假日：specialDays 为 2026 年硬编码，明年需手工维护或接后台/第三方日历 API。
- 积分/智体入口：pointStats、campusTools 部分仍为静态/模拟数据，与真实待办无关。

5. 补 2～3 条关键路径测试（切换月份、完成/撤销、通知接受）。
