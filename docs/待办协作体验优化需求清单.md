# 待办协作体验优化需求清单

> 文档版本：v0.1  
> 创建日期：2026-07-14  
> 当前阶段：需求讨论与后台能力确认  
> 适用范围：AITodo 首页待办、日历、消息中心及智能待办后台接口

## 1. 文档用途

本文档作为待办协作优化的需求主文档，用于：

- 产品侧确认实际使用规则和优先级。
- 前端与后台逐项确认数据结构、接口行为和权限边界。
- 开发过程中记录需求状态、待确认问题和变更结论。
- 联调和验收时作为统一检查清单。

本文档记录的是期望业务结果。文中出现的接口名称和字段名称属于建议方案，最终以后端确认结果为准。

## 2. 推荐协作方式

### 2.1 主文档

以本 Markdown 文件作为项目内的唯一需求基线，和代码一起进行版本管理。重要规则发生变化时，应同步修改本文档，避免聊天记录和实际开发要求不一致。

### 2.2 日常讨论

可以把本文档同步到飞书文档，用于评论、@相关人员和会议讨论。讨论完成后的最终结论仍应回写到本文件。

### 2.3 执行进度

事项较少时，直接维护本文档中的需求总表即可。事项增多后，建议建立飞书多维表格，每一行对应本文档中的一个需求 ID，并至少包含以下字段：

- 需求 ID
- 需求名称
- 优先级
- 产品结论
- 后台负责人
- 前端负责人
- 当前状态
- 计划完成时间
- 接口文档地址
- 联调结果
- 验收结果
- 补充说明

建议统一使用以下状态：

`待确认 → 已确认 → 开发中 → 待联调 → 联调中 → 待验收 → 已完成`

特殊状态：`暂缓`、`不做`、`有阻塞`。

## 3. 角色与术语

| 术语 | 说明 |
| --- | --- |
| 派发人 | 创建并派发待办的用户 |
| 接受人 | 被指派处理待办的用户 |
| 主待办 | 派发人侧用于管理一次派发任务的主体记录 |
| 子待办 | 与某个接受人对应的执行记录，保存其接受、拒绝、进展和完成状态 |
| 有效待办 | 当前用户仍有查看或操作权限的待办 |
| 撤销 | 派发人终止协作待办，保留必要的操作记录和通知 |
| 删除 | 清理个人数据或进行软删除；协作待办原则上不直接物理删除 |
| 归档/隐藏 | 仅从当前用户常用列表移除，不影响其他参与人和后台业务数据 |
| 过期 | 当前时间超过待办有效截止时间且待办仍未进入终态；建议作为计算属性，不替代业务状态 |

## 4. 总体产品原则

1. 业务关系是否有效，不能由消息是否已读决定。
2. 多人待办必须保留主待办与各接受人子待办之间的稳定关联。
3. 修改接受人时采用差量更新，保留未变化接受人的状态和进展。
4. 权限判断以后台为准；关键身份字段缺失时，前端应只读处理，不能放宽权限。
5. 所有入口执行同一业务操作后，日历、待办列表、详情、消息和角标必须得到相同结果。
6. 协作待办优先采用撤销和软删除，避免丢失责任记录。
7. 消息已读、待办待响应和待办业务状态是三个不同概念，应分别维护。

## 6. P0 需求明细

### TODO-001 编辑接受人及差量更新

#### 需求背景

派发人编辑待办时，需要能够增加、保留或移除接受人。当前项目已经存在多人接受人和主、子待办结构，但更新接口对多人接受人的处理规则尚未形成明确闭环。

#### 目标规则

假设修改前接受人为 A、B，修改后为 B、C：

- A 属于被移除接受人：立即撤销其业务权限，立即从其有效待办列表中移除，并发送变更通知。
- B 属于保留接受人：保留原子待办 ID、接受状态、完成状态、进展记录和操作历史。
- C 属于新增接受人：创建新的子待办，进入待接受状态，并发送新派发通知。
- 消息是否已读只影响消息状态，不决定 A 的待办何时失效。
- 如果 A 正在查看详情，收到变更事件后应提示“你已不再是该待办接受人”，并关闭或切换为无操作权限的快照。
- 接受人列表不允许因为更新而出现重复人员。
- 是否允许派发人把所有接受人都移除，需要产品和后台确认。建议此时将协作待办转换为派发人个人待办，或要求至少保留一名接受人，不能产生无归属的异常数据。

#### 后台待确认

- `PUT /smart-todo` 更新多人接受人时应使用 `assigneeIds` 还是 `assigneeId`。
- 后台是否按主待办 ID 自动计算新增、保留和移除集合。
- 被移除的子待办使用什么状态或删除标记。
- 被移除接受人的历史进展是否继续保留给派发人查看。
- 更新是否在一个数据库事务内完成。
- 返回值是否可以返回更新后的完整主待办、子待办列表及变更人员集合。

#### 验收标准

- 修改接受人后，保留人员的原状态和进展不丢失。
- 新增人员只收到一次派发通知，并出现在待接受列表。
- 被移除人员立即无法继续完成、拒绝或提交进展。
- 被移除人员收到一次说明清楚的通知。
- 页面刷新、重新登录和 WebSocket 断线重连后结果保持一致。

### TODO-002 派发人撤销后同步所有接受人

#### 需求背景

派发人删除或撤销协作待办后，所有接受人侧的关联数据需要同步失效，并收到通知。

#### 目标规则

- 个人待办可以删除；已经派发给他人的协作待办，操作名称建议使用“撤销待办”。
- 派发人撤销主待办时，后台在同一事务内撤销所有仍有效的子待办。
- 当前有效接受人收到“待办已被派发人撤销”的消息。
- 接受人的日历、待办列表、待接受列表和已打开详情立即同步。
- 已经在此前被移除的接受人不重复收到本次撤销通知。
- 撤销通知保留必要的待办快照，但不能继续跳转到不存在的普通详情；也可以提供只读的撤销记录页。
- 已完成、已拒绝或正在处理的协作待办是否允许撤销，需要产品确认。建议允许撤销但必须二次确认，并记录撤销原因。

#### 后台待确认

- 当前 `DELETE /smart-todo/{id}` 是物理删除、软删除，还是会级联处理子待办。
- 是否新增独立的撤销接口，例如 `POST /smart-todo/revoke`。
- 撤销后主待办和子待办分别使用什么状态。
- 撤销消息如何携带原待办标题、内容、派发人、撤销时间和撤销原因。
- 被撤销的记录保留多长时间。

#### 验收标准

- 派发人撤销成功后，所有接受人立即失去操作权限。
- 所有当前有效接受人各收到一次通知。
- 任一接受人重新登录后不会再次看到有效待办。
- 撤销消息不会出现点击后查询普通详情失败的死链。
- 主待办和全部子待办不存在部分成功、部分失败的中间状态。

### TODO-003 权限规则收紧和显式操作权限

#### 当前风险

- 当前前端在 `creatorId` 缺失时可能仍推断用户可以编辑。
- 已完成或已拒绝待办的参与人可能看到删除入口，而当前删除接口可能影响整条协作待办。
- `currentHandlerId`、`receiveStatus` 等字段缺失时，前端推断结果可能与真实权限不一致。

#### 目标规则

- 只有派发人可以编辑和撤销整条协作待办。
- 只有当前有效接受人可以接受、拒绝、提交进展、完成或转交自己的子待办。
- 接受人清理个人列表时只能归档或隐藏，不能调用全局删除。
- 已被移除或已撤销的接受人只能查看通知或历史快照。
- 后台必须对每一次写操作重新校验权限，不能只依赖前端隐藏按钮。
- 关键权限字段缺失时，前端默认只读并提示数据不完整。

#### 建议返回字段

后台详情和列表可以直接返回：

```json
{
  "canEdit": false,
  "canRevoke": false,
  "canDelete": false,
  "canAccept": true,
  "canReject": true,
  "canComplete": false,
  "canReopen": false,
  "canAddProcess": false,
  "canTransfer": false,
  "canArchive": true
}
```

字段名称可以调整，但权限含义需要明确。

#### 验收标准

- 伪造前端请求也无法越权修改或删除他人待办。
- `creatorId/currentHandlerId` 缺失时不会出现可编辑或可删除按钮。
- 接受人归档自己的记录不会影响派发人和其他接受人。

### TODO-004 完整待办状态机

#### 当前问题

后台状态 `0` 同时承担待处理和待接受含义，前端主要使用 `todo/done` 展示状态。待接受、已接受、已拒绝、已撤销之间的操作边界不够明确。

#### 建议状态与操作

| 生命周期状态 | 接受人可操作 | 派发人可操作 | 列表表现 |
| --- | --- | --- | --- |
| 待接受 | 接受、拒绝 | 编辑、撤销 | 接受人进入待响应，派发人显示等待接受 |
| 进行中 | 提交进展、完成、申请转交 | 编辑、催办、撤销 | 双方正常可见 |
| 已完成 | 查看；是否可恢复待确认 | 查看、归档；是否可恢复待确认 | 进入已完成历史 |
| 已拒绝 | 查看拒绝原因 | 修改后重新派发、归档 | 不再作为接受人的有效任务 |
| 已撤销 | 查看通知或快照 | 查看记录、归档 | 不再作为有效任务 |
| 已删除 | 无 | 无 | 默认不展示，仅审计数据保留 |

“已过期”建议作为派生属性：截止时间已过且生命周期状态不是已完成、已拒绝、已撤销或已删除。它不应覆盖原来的生命周期状态。

#### 后台待确认

- 现有 `status=0/3/6/9/99` 和 `receiveStatus` 的准确组合含义。
- 是否需要新增独立的已撤销状态。
- 多接受人情况下，主待办状态如何根据子待办汇总。
- 部分完成、部分拒绝、部分待接受时主待办显示什么状态。
- 已完成待办由谁恢复，恢复后回到什么状态。
- 待接受状态是否允许直接完成。建议不允许。

#### 验收标准

- 每一种状态的按钮和权限固定、可解释。
- 待接受人员不能直接提交进展或完成。
- 已拒绝和已撤销待办不会继续显示为普通待处理。
- 多人待办可以准确显示各接受人的独立状态和整体完成进度。

### TODO-005 不同操作入口结果一致

#### 当前问题

用户可以从消息中心、日历列表、详情面板和不同工作台模式执行接受、拒绝、完成等操作。当前不同入口执行后的消息已读、列表刷新和拒绝原因处理不完全一致。

#### 目标规则

前端应建立统一的待办操作流程。任一入口执行成功后，都应按操作类型同步：

- 当前日历范围。
- 待接受列表。
- 消息列表和消息已读状态。
- 未读角标与待响应角标。
- 当前打开的待办详情。
- 主待办和子待办进度汇总。

拒绝待办时应使用统一的原因输入流程，不能某些入口允许填写原因，另一些入口固定保存“暂不处理”。

#### 验收标准

- 从任何入口接受后，待接受角标和相关通知结果一致。
- 从任何入口拒绝后，派发人看到相同格式的拒绝原因。
- 从任何入口完成后，所有列表和详情状态一致。
- 简约模式和详细模式结果一致。

### TODO-006 实时消息驱动待办同步

#### 当前问题

当前 WebSocket 消息主要刷新站内消息和待接受列表，日历、当前详情和其他状态可能继续保留旧数据。

#### 建议事件类型

- `todo_created`
- `todo_updated`
- `todo_assignee_added`
- `todo_assignee_removed`
- `todo_accepted`
- `todo_rejected`
- `todo_completed`
- `todo_reopened`
- `todo_revoked`
- `todo_deleted`
- `todo_overdue`
- `todo_process_added`

建议消息至少包含：业务 ID、主待办 ID、子待办 ID、动作类型、变更版本、发生时间、是否允许打开详情，以及必要的展示快照。

#### 前端规则

- 收到待办相关事件后，静默刷新受影响的日历、待接受列表、消息和详情。
- 页面重新获得焦点或从后台恢复时，进行一次静默增量刷新。
- WebSocket 断线重连后重新拉取最新状态，不能只依赖断线期间的推送补发。
- 当前详情已失效时，关闭操作区并提示具体原因。

#### 验收标准

- 两个用户同时在线时，一方操作后另一方无需手动刷新即可看到结果。
- 断网后恢复连接，数据最终与后台一致。
- 同一事件重复推送不会产生重复通知或重复列表项。

### TODO-007 编辑表单字段与接口一致

#### 当前问题

- 编辑界面允许修改待办类型，但当前更新请求未明确提交 `type`。
- 创建多人待办使用 `assigneeIds`，更新使用 `assigneeId`，多人更新语义需要统一。
- `title` 和 `content` 在 AI 创建及普通编辑中的含义不同，需要避免更新时覆盖错误字段。

#### 目标规则

- 页面允许编辑的字段必须能够真实保存。
- 后台不支持编辑的字段，前端暂时禁用并说明原因。
- 创建和更新接口对标题、内容、类型、时间、备注和接受人的字段定义保持一致。
- 更新成功最好返回完整新数据，前端直接以后台结果更新页面。

#### 后台待确认字段

- `title`：AI 原始指令还是用户可见标题。
- `content`：用户可见待办内容还是详细说明。
- `remark`：备注。
- `type`：待办或会议，是否允许后续修改。
- `assigneeId/assigneeIds`：单人和多人更新规则。
- `version/updateTime`：并发更新版本。

#### 验收标准

- 编辑后刷新页面，所有已保存字段与编辑结果一致。
- 不出现页面提示保存成功但字段实际未变化。
- 多人接受人修改不会丢失未变化人员的执行状态。

### TODO-008 待办过期提醒

#### 目标规则

- 单时间待办以 `startDateShow` 作为计划时间；时间范围待办以 `endDateShow` 作为截止时间。
- 日期型全天待办的默认截止时刻建议为当天 `23:59:59`。
- 当前时间超过截止时间，且待办不是已完成、已拒绝、已撤销或已删除时，判定为过期。
- 已接受但未完成：给接受人发送一次过期提醒。
- 到期时仍未接受：给接受人发送“待接受事项已到期”提醒，并可同时提醒派发人。
- 可选规则：过期 24 小时仍未完成时，再提醒派发人一次。
- 同一待办、同一接受人、同一截止时间版本、同一提醒类型只能生成一次消息。
- 修改截止时间后，旧提醒计划失效，按新截止时间重新计算。
- 完成、拒绝、移除接受人或撤销待办后，取消后续提醒。
- 提醒必须由后台定时任务生成并保存为站内消息，不能只依赖用户在线时的前端定时器。

#### 后台待确认

- 是否已有定时任务框架。
- 消息去重键和任务补偿机制。
- 服务重启后如何补发应发但未发的提醒。
- 过期提醒是否同时推送到桌面宠物未读角标。
- 派发人二次提醒时间是否固定为 24 小时或可配置。

#### 验收标准

- 用户不在线时过期，重新登录仍能收到消息。
- 同一截止版本不会重复提醒。
- 修改截止时间、完成或撤销后不会继续收到旧提醒。
- 点击提醒能够打开有效详情；已撤销时显示快照而不是报错。

### TODO-009 日期、时间和全天待办规则

#### 目标规则

- 支持只有日期、没有具体时间的全天待办。
- 同一天的范围待办，截止时间必须晚于开始时间。
- 跨天待办，截止日期不能早于开始日期。
- 创建过去时间的待办时进行明确确认，不应静默保存。
- AI 解析出的时间不合法或已过期时，要求用户确认或修正。
- 创建和编辑使用同一套校验规则。
- 前后台统一采用 `Asia/Shanghai` 业务时区，接口明确是否传本地时间或带时区时间。
- 是否允许凌晨 `00:00-06:59` 的任务应由产品确认；如果允许，时间选择器必须提供。

#### 验收标准

- 无法保存截止早于开始的范围。
- 全天待办可以正常创建、编辑、显示和提醒。
- 创建与编辑对相同数据给出相同校验结果。
- AI 解析不会生成用户无法察觉的过期事项。

### TODO-010 全部未完成与逾期待办聚合视图

#### 需求背景

日历按月或周加载时，上个月遗留的未完成待办容易离开当前视野。过期提醒也需要一个稳定的统一处理入口。

#### 建议视图

- 我的未完成。
- 我派发但未完成。
- 已逾期。
- 待接受。
- 已拒绝。
- 最近完成。

#### 后台建议

提供独立于日历范围的分页查询，支持按当前用户、派发方向、生命周期状态、是否过期、类型和关键词筛选。具体接口名称待确认，例如：

`GET /smart-todo/active-list`

#### 验收标准

- 上月或更早的未完成待办可以在聚合视图找到。
- 过期提醒点击后能定位到对应待办。
- 列表支持分页，不因历史数据增加而明显变慢。

### TODO-011 未读消息与待响应事项分离

#### 当前问题

消息中心角标同时计算未读站内消息和待接受待办，但“全部标记已读”只能改变消息状态，无法处理待接受事项，因此用户点击后角标可能仍然存在。

#### 目标规则

- 未读消息：可以单条或全部标记已读。
- 待响应事项：必须接受、拒绝或完成对应操作后才从待响应中移除。
- 总角标可以继续合并显示，但消息中心内部需要分别展示数量和含义。
- “全部标记已读”按钮只根据未读消息数量启用，不能因为存在待接受事项而启用。
- 待接受事项与其关联消息继续去重，避免同一业务算两次。

#### 验收标准

- 只有待接受事项时，“全部标记已读”不会产生无效果操作。
- 接受或拒绝后，待响应数量立即减少。
- 标记消息已读不会错误改变待办状态。

### TODO-012 拒绝原因和操作时间线

#### 目标规则

- 拒绝操作统一弹出原因输入；是否必填由产品确认，建议至少要求选择一个预设原因或填写说明。
- 派发人可以在详情中看到每个接受人的拒绝原因和时间。
- 每条协作待办保留操作时间线，至少记录：
  - 创建和派发。
  - 接受或拒绝。
  - 接受人增加或移除。
  - 标题、内容、时间和备注修改。
  - 进展提交和修改。
  - 完成和恢复。
  - 转交、催办和撤销。
- 时间线记录应包含操作人、时间、动作和必要的变更摘要。
- 审计记录不能由普通用户直接删除或篡改。

#### 验收标准

- 从任何入口拒绝都能保存相同格式的原因。
- 派发人能明确知道谁在何时因何原因拒绝。
- 关键协作变更可以追溯。

### TODO-013 异常降级、幂等和并发冲突保护

#### 目标规则

- 人员列表加载失败时，待办列表仍应正常加载，人员名称可暂时回退为工号。
- 创建待办和关键状态操作支持幂等，避免请求超时后用户重试产生重复待办或重复通知。
- 编辑请求携带 `version` 或 `updateTime`。后台发现数据已被其他用户修改时返回冲突，前端提示用户重新加载后再保存。
- 乐观更新失败后恢复后台真实状态。
- 接口重复提交不会重复创建子待办、进展或消息。

#### 后台待确认

- 是否支持 `Idempotency-Key/requestId`。
- 是否支持版本号或更新时间并发控制。
- 冲突时统一使用什么错误码和响应结构。
- 消息和定时提醒的去重机制。

#### 验收标准

- 人员接口故障不会让整个日历变为空白。
- 网络超时后重试不会产生两条相同待办。
- 派发人编辑与接受人完成同时发生时，不会静默覆盖对方结果。

## 9. 后台接口确认清单

与后台沟通时建议逐项确认以下问题，并把结论回写到本文档。

### 9.1 数据模型

- 主待办和子待办的 ID、`fid` 关系是否稳定。
- 多接受人每人是否始终对应独立子待办。
- `status` 与 `receiveStatus` 的全部枚举和组合含义。
- 被移除、已撤销、已归档和已删除分别如何存储。
- 主待办的汇总状态如何计算。
- 进展和操作时间线保留策略。

### 9.2 权限

- 派发人、当前接受人、已移除接受人分别可以执行哪些接口。
- `user-list` 是否只返回当前用户有权选择的人员。
- 是否由后台返回显式操作权限字段。
- 身份字段缺失时接口如何处理。

### 9.3 更新接口

- 更新多人接受人使用哪个字段。
- 更新时是否支持 `type`。
- `title/content/remark` 的准确含义。
- 更新是否返回完整最新实体。
- 是否支持版本冲突检测。

### 9.4 删除与撤销

- 当前删除接口是否级联子待办。
- 是否需要新增撤销接口和撤销原因。
- 已完成、已拒绝、进行中的协作待办能否撤销。
- 接受人“从我的列表删除”应该对应归档还是用户级隐藏。

### 9.5 消息和 WebSocket

- 是否能够增加明确的待办动作类型。
- 消息是否包含业务版本和是否可打开详情。
- 撤销或删除消息如何提供快照。
- WebSocket 断线期间消息是否补发。
- 重复推送和重复消息如何去重。

### 9.6 过期提醒

- 定时任务和失败补偿方案。
- 过期判定时区。
- 去重键。
- 截止时间修改后的旧任务取消方式。
- 是否同时提醒派发人。

### 9.7 会议类型

- `type=2` 的会议是否完整复用待办的接受、拒绝、完成、撤销和过期逻辑。
- 如果会议规则不同，需要明确哪些接口、状态和通知不能复用。

## 10. 关键联调场景

### 场景 A：修改多人接受人

1. 派发人将接受人从 A、B 修改为 B、C。
2. 确认 A 立即失去权限并收到通知。
3. 确认 B 的状态和进展保持不变。
4. 确认 C 进入待接受并收到一次通知。
5. 刷新、重登和断线重连后再次确认结果。

### 场景 B：撤销协作待办

1. 派发人撤销一条包含多个接受人的待办。
2. 确认所有有效接受人立即移除该待办并收到通知。
3. 确认通知可以正常查看快照，不出现详情查询失败。
4. 确认已被提前移除的人员不重复收到通知。

### 场景 C：过期提醒

1. 创建短时间截止待办并保持未完成。
2. 确认到期后只生成一次提醒。
3. 修改截止时间并确认旧提醒计划失效。
4. 完成或撤销后确认不再提醒。
5. 用户离线到期后重新登录，确认消息仍存在。

### 场景 D：不同入口接受和拒绝

1. 分别从消息中心、日历详情和两种工作台模式操作。
2. 确认待办状态、相关消息、角标和列表刷新结果一致。
3. 确认拒绝原因格式一致。

### 场景 E：权限字段缺失

1. 模拟列表或详情缺少 `creatorId/currentHandlerId/receiveStatus`。
2. 确认页面只读且不会出现越权操作按钮。
3. 直接调用写接口，确认后台拒绝越权请求。

### 场景 F：实时同步和断线恢复

1. 派发人和接受人同时在线。
2. 一方编辑、完成或撤销，确认另一方无需手动刷新即可看到变化。
3. 断开接受人网络后继续操作，恢复网络后确认最终数据一致。

### 场景 G：并发修改

1. 派发人打开编辑页，接受人同时完成待办。
2. 派发人提交旧版本编辑。
3. 确认系统提示数据已变化，而不是覆盖完成状态。

## 11. 完成定义

- 产品规则已经确认并记录。
- 后台接口和数据结构已经确认。
- 后台开发完成并通过接口自测。
- 前端开发完成并通过代码检查和自动化测试。
- WebSocket、消息、日历、详情和多用户场景完成联调。
- 本文档对应验收标准全部通过。
- 已知限制和暂缓项已记录，不以口头约定代替。
