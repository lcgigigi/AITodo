# 备忘录功能 — 后台接口方案（精简版）

> 文档版本：v2.0  
> 更新日期：2026-07-23  
> 原则：**复用 `/smart-todo/*`，用字段区分备忘录与任务流；尽量少新增 HTTP 路径**  
> 关联：[备忘录功能-前端实现方案](./备忘录功能-前端实现方案.md) · [smart-todo-api-integration.md](./smart-todo-api-integration.md)

---

## 1. 目标

内网同事间传递链接、账号、说明等碎片化信息。备忘录 **不上日历、无完成态、无逾期**，支持 1 对 1 派发 → 接受 → 钉首页。

**不做**：独立 `/smart-memo/*` 模块、备忘录上日历、多人派发、前端展示修改历史（后台需落库）。

---

## 2. 已确认规则

| # | 规则 |
| --- | --- |
| 1 | 派发后发送方可改；后台写修改记录，前端首期不展示 |
| 2 | 接收方可拒绝；拒绝后有效列表不展示，通知发送方 |
| 3 | 「删除」改「归档」；提供已归档列表 |
| 4 | 备忘录与待办/会议的已归档 **同一接口、不同 query 区分** |
| 5 | 仅 1 对 1 派发 |
| 6 | 接受后默认 `pinned=1` |

---

## 3. 核心设计：一张表 + 一个主字段

在现有待办表上扩展字段，**不新建业务模块**。

### 3.1 新增 / 扩展字段

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| **`recordMode`** | `1` \| `2` | **`1`=任务流（默认）**，`2`=备忘录。**主区分字段** |
| **`pinned`** | `0` \| `1` | 是否钉在首页 |
| **`archivedFlag`** | `0` \| `1` | 是否归档（与 `status=99` 二选一，项目内写死一种） |

沿用现有字段：

| 字段 | 备忘录用法 |
| --- | --- |
| `title` / `content` | 标题 / 正文 |
| `assigneeId` | 接收人（1 对 1）；空 = 仅自己 |
| `status` | `0` 待接受 · `3` 已接受 · `9` 已拒绝 · **`6` 不使用** |
| `type` | 可固定 `1`，或增 `3=备忘录`（与 `recordMode` 二选一即可，**推荐只认 recordMode** |
| `timeType` / 日期时间 | 可选；仅展示，不参与逾期 |

### 3.2 修改记录（仅落库，首期无 HTTP）

表 `smart_todo_revision`（或等价）：`todoId`、`editorId`、变更前后 title/content、`createTime`。  
每次 `PUT /smart-todo` 且 `recordMode=2` 时写入。

## 4. 接口策略总览

```text
复用（加字段 / 加分支）     扩展 query 或 +1～2 个 path
─────────────────────     ───────────────────────────
POST   /smart-todo/create
PUT    /smart-todo
DELETE /smart-todo/{id}      → 语义改为归档
GET    /smart-todo/{id}
POST   /smart-todo/accept
POST   /smart-todo/reject
GET    /smart-todo/pending-list
GET    /smart-todo/month-list      → 默认 recordMode=1
GET    /smart-todo/today-list      → 默认 recordMode=1

新增（建议仅 2 个）
GET    /smart-todo/memo-list?scope=...
GET    /smart-todo/archived-list?recordMode=1|2

禁止（recordMode=2 时返回 400）
POST   /smart-todo/complete/{id}
进展 / process 相关写入
```

## 5. 复用接口说明

### 5.1 创建 `POST /smart-todo/create`

请求体在现有基础上增加：

```json
{
  "title": "测试环境地址",
  "content": "https://intranet.example.com/app",
  "recordMode": 2,
  "assigneeIds": "1102080",
  "pinned": 1,
  "type": 1,
  "timeType": 1,
  "startDateShow": "",
  "remark": ""
}
```

| 规则 | 说明 |
| --- | --- |
| `recordMode=2` 必填 | 标识备忘录 |
| 无接收人 | `status` 视为仅自己；创建人可 `pinned=1` |
| 有接收人 | `status=0`；发 sys_message + WebSocket（`bizType=3`） |
| 时间 | 可空；不参与逾期 |

### 5.2 更新 `PUT /smart-todo`

- 创建人可改 `title`、`content`、`pinned`
- **MVP 不可改 `assigneeId`**
- 写入 revision 表；已接受的派发建议通知接收方「内容已更新」
- 请求体带 `recordMode: 2`（或后台按 id 校验不可改 mode）

### 5.3 归档 `DELETE /smart-todo/{id}`

- 语义改为 **软归档**（`archivedFlag=1` 或 `status=99`，项目内统一）
- 有效列表、日历、memo-list 均不再返回
- 可选另增 `POST /smart-todo/unarchive/{id}` 恢复（非必须）

### 5.4 接受 / 拒绝

与现有一致：

- `POST /smart-todo/accept` — 成功后 **`pinned=1`**
- `POST /smart-todo/reject` — 写 `handleDesc` 作拒绝原因

### 5.5 详情 `GET /smart-todo/{id}`

返回体增加 `recordMode`、`pinned`、`archivedFlag`。前端据 `recordMode` 决定 UI。

### 5.6 待接受 `GET /smart-todo/pending-list`

**默认或加参** `recordMode=2`，供备忘录 inbox；待办 pending 传 `recordMode=1`。  
若不便改现有行为，可约定：pending-list 返回全部，前端按 `recordMode` 分流。

### 5.7 日历 / 今日列表（必改过滤）

| 接口 | 规则 |
| --- | --- |
| `GET /smart-todo/month-list` | **仅 `recordMode=1` 且未归档** |
| `GET /smart-todo/today-list` | **仅 `recordMode=1` 且未归档** |

备忘录 **不得** 出现在上述结果中。

### 5.8 完成（备忘录禁用）

`POST /smart-todo/complete/{id}`：当 `recordMode=2` 时 **400**，提示「备忘录不支持完成操作」。

## 6. 新增接口（仅 2 个）

### 6.1 备忘录列表 `GET /smart-todo/memo-list`

一个接口覆盖备忘录页 + 首页钉住区。

**Query**

| 参数 | 值 | 用途 |
| --- | --- | --- |
| `scope` | `created` | 我创建的（未归档） |
| | `received` | 我收到的（未归档） |
| | `pending` | 待我接受 |
| | `pinned` | 已钉住（首页条） |
| | `archived` | 已归档（等同 archived-list + recordMode=2，可二选一） |
| `page` / `pageSize` | | 分页 |

**固定条件**：`recordMode=2`。

**pinned 示例响应字段**：`id, title, content, creatorName, pinned, status, updateTime`。

### 6.2 已归档列表 `GET /smart-todo/archived-list`

**Query**

| 参数 | 说明 |
| --- | --- |
| **`recordMode=2`** | 备忘录已归档（备忘录页 Tab） |
| **`recordMode=1`** | 待办/会议已归档（待办侧入口） |
| `page` / `pageSize` | 分页 |

同一 path，**靠 query 区分**，不混在一个列表里返回。

## 7. 站内消息 / WebSocket

扩展 `bizType`：

| bizType | 含义 |
| --- | --- |
| 1 | 智能待办 |
| 2 | 会议 |
| **3** | **备忘录**（`bizId` 仍为 smart_todo 表 id） |

场景：派发、接受、拒绝、内容更新。推送格式不变（`type: sys_message`）。

---

## 8. 后台分支规则（防漏）

所有 **任务向** 查询默认加：`recordMode = 1 AND archivedFlag = 0`（或等价条件）。

| 场景 | recordMode=2 行为 |
| --- | --- |
| month-list / today-list | 不返回 |
| complete | 拒绝 |
| processList | 不写入 / 不返回 |
| 过期提醒定时任务 | 跳过 |
| accept | 允许，`pinned=1` |
| 归档 | 允许 |

---

## 9. 归档与协作待办（简版）

| 对象 | 归档 |
| --- | --- |
| 备忘录 | 创建人 / 接收人各自归档（推荐用户级 `archivedFlag`）；MVP 可先做全局归档 |
| 个人待办 | 创建人归档 → `archived-list?recordMode=1` |
| 协作待办 | 接受人：个人归档；派发人：建议仍用 **撤销** 而非归档（产品另议） |
