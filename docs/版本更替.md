# 版本规划文档

本文档用于记录当前线上版本与后续版本的功能规划，便于产品、前端、后端和测试统一理解版本边界。

## 1. 当前线上版本

当前线上版本为 `v1`。

`v1` 已支持事项的基础创建、派发、查看和状态流转能力，核心目标是完成待办事项从创建到处理的基础闭环。

## 2. 后续版本规划

后续版本为 `v2`。

`v2` 计划在事项协作能力上继续增强，重点支持派发人与接收人围绕具体事项进行沟通交互。

## 3. v2 功能：事项沟通交互

### 3.1 功能目标

派发人和接收人可以在具体事项内进行沟通，沟通内容与事项绑定，避免事项处理过程中的信息分散在外部聊天工具或口头沟通中。

该能力上线后，用户可以在事项详情中查看完整沟通记录，并基于沟通内容推进事项处理。

### 3.2 适用角色

| 角色 | 说明 |
| --- | --- |
| 派发人 | 创建并派发事项的用户 |
| 接收人 | 被指派处理事项的用户 |

### 3.3 核心场景

| 场景 | 说明 |
| --- | --- |
| 派发人补充说明 | 派发人可以在事项中追加背景、要求、截止说明等信息 |
| 接收人反馈进展 | 接收人可以在事项中反馈处理进度、问题和完成情况 |
| 双方确认事项 | 派发人与接收人可以基于沟通记录确认需求、调整预期或补充材料 |
| 留存处理过程 | 所有沟通内容保留在事项详情中，便于后续追溯 |

### 3.4 功能范围

`v2` 建议包含以下能力：

- 事项详情中展示沟通记录。
- 派发人和接收人可以发送文本消息。
- 沟通记录按时间顺序展示。
- 每条消息展示发送人、发送时间和消息内容。
- 仅事项相关人员可查看和发送沟通消息。

### 3.5 权限规则

| 操作 | 派发人 | 接收人 | 其他用户 |
| --- | --- | --- | --- |
| 查看沟通记录 | 可操作 | 可操作 | 不可操作 |
| 发送沟通消息 | 可操作 | 可操作 | 不可操作 |
| 删除沟通消息 | 暂不支持 | 暂不支持 | 不可操作 |

### 3.6 数据字段建议

```ts
interface TodoComment {
  id: string
  todoId: string
  senderId: string
  senderName: string
  content: string
  createdAt: string
}
```

字段说明：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | string | 沟通消息 ID |
| todoId | string | 所属事项 ID |
| senderId | string | 发送人 ID |
| senderName | string | 发送人名称 |
| content | string | 消息内容 |
| createdAt | string | 发送时间 |

### 3.7 接口建议

#### 查询事项沟通记录

`GET /api/todos/{id}/comments`

返回指定事项下当前用户可见的沟通记录。

#### 发送事项沟通消息

`POST /api/todos/{id}/comments`

请求：

```json
{
  "content": "这个事项需要补充一下验收标准。"
}
```

规则：

- 当前用户必须是事项派发人或接收人。
- `senderId`、`senderName`、`createdAt` 由后端根据登录态和服务器时间生成。
- 消息内容不能为空。

## 4. 验收标准

- 派发人进入事项详情后，可以查看并发送沟通消息。
- 接收人进入事项详情后，可以查看并发送沟通消息。
- 非事项相关人员无法查看或发送该事项的沟通消息。
- 新发送的消息可以立即出现在事项沟通记录中。
- 刷新页面后，历史沟通记录仍然存在。
