# 待办实时同步优化方案（后台）

> 文档版本：v1.0  
> 创建日期：2026-07-22  
> 适用范围：智能待办（smart-todo）、站内消息（sys-message）、WebSocket 推送服务  
> 关联文档：[完整版](./待办实时同步优化方案-完整版.md) · [前端方案](./待办实时同步优化方案-前端.md)  
> 现有接口参考：[sys-message-websocket接口说明.txt](./sys-message-websocket接口说明.txt) · [smart-todo-api-integration.md](./smart-todo-api-integration.md)

---

## 1. 目标

后台建立 **变更事件驱动** 的数据同步能力，使前端在多人协作场景下无需手动刷新页面即可看到最新待办/会议数据。

核心交付：

1. 所有待办/会议写操作成功后发布标准 `todo_changed` 事件
2. 通过现有 WebSocket 通道定向推送给相关用户
3. 提供 `changes` 增量补偿接口，支持断线对账
4. 提供版本号机制，支持并发冲突检测

---

## 2. 现状

### 2.1 已有能力

| 能力 | 说明 |
| --- | --- |
| WebSocket | `ws(s)://{host}/websocket/{userId}`，按用户工号建连 |
| 推送格式 | 当前仅 `type: sys_message` |
| sys_message | 含 `bizType`（1=待办，2=会议）、`bizId` |
| 待办 CRUD | `/smart-todo/*` 系列 REST 接口 |
| 列表查询 | `GET /smart-todo/month-list?startDate&endDate` |

### 2.2 缺口

| 缺口 | 影响 |
| --- | --- |
| 写操作不统一发数据变更事件 | 前端只能靠轮询或 sys_message 间接感知 |
| sys_message 与数据变更耦合 | 有些变更可能无消息，有些消息不保证数据完整 |
| 无 changes 补偿接口 | 断线期间事件丢失无法恢复 |
| 无 version 字段 | 多人同时编辑可能互相覆盖 |
| 推送对象不完整 | 仅消息接收人收到推送，协作者可能遗漏 |

## 4. 变更事件设计

### 4.1 事件模型：`TodoChangedEvent`

```json
{
  "type": "todo_changed",
  "eventId": "evt_20260722_8f3a2b1c",
  "action": "updated",
  "entityType": "todo",
  "todoId": "12345",
  "mainTodoId": "12345",
  "childTodoId": "12346",
  "affectedUserIds": ["10001", "10002", "10003"],
  "dateRange": {
    "startDate": "2026-07-01",
    "endDate": "2026-07-31"
  },
  "version": 7,
  "updatedAt": "2026-07-22T15:40:00+08:00",
  "canOpenDetail": true,
  "snapshot": {}
}
```

### 4.2 字段规范

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `type` | string | 是 | 固定 `todo_changed` |
| `eventId` | string | 是 | 全局唯一，建议 `{date}_{uuid}` |
| `action` | string | 是 | 见 4.3 action 枚举 |
| `entityType` | string | 是 | `todo` / `meeting` |
| `todoId` | string | 是 | 前端用于列表/详情的主键 |
| `mainTodoId` | string | 否 | 主待办 ID |
| `childTodoId` | string | 否 | 子待办 ID（多人接受场景） |
| `affectedUserIds` | string[] | 是 | 需推送的用户工号 |
| `dateRange.startDate` | string | 是 | `YYYY-MM-DD` |
| `dateRange.endDate` | string | 是 | `YYYY-MM-DD` |
| `version` | number | 推荐 | 实体版本号，每次写操作 +1 |
| `updatedAt` | string | 是 | ISO 8601，业务时区建议 `+08:00` |
| `canOpenDetail` | boolean | 推荐 | 撤销/删除后设为 false |
| `snapshot` | object | 否 | 变更后实体，结构与 month-list 单项一致 |

### 4.3 action 枚举

| action | 触发接口/场景 |
| --- | --- |
| `created` | `POST /smart-todo/create`、`POST /smart-todo/sync-calendar` |
| `updated` | `PUT /smart-todo` |
| `deleted` | `DELETE /smart-todo/{id}` |
| `status_changed` | accept / reject / complete / cancel |
| `assignee_added` | 更新接受人时新增人员 |
| `assignee_removed` | 更新接受人时移除人员 |
| `revoked` | `POST /smart-todo/cancel`（撤销协作待办） |
| `process_added` | `POST /smart-todo/process` |

### 4.4 affectedUserIds 计算规则

推送对象 = 以下去重并集：

| 角色 | 是否推送 |
| --- | --- |
| 创建者（creatorId） | 是 |
| 当前接受人/执行人（assigneeId / currentHandlerId） | 是 |
| 所有子待办的处理人 | 是 |
| 被新增/移除的接受人 | 是 |
| 转交前/后的接受人 | 是 |
| 与待办无关联的其他用户 | 否 |

> 注意：被移除的接受人也应收到 `assignee_removed` 事件，以便前端立即移除列表项。

### 4.5 dateRange 计算规则

```
startDate = min(变更前 startDate, 变更后 startDate)
endDate   = max(变更前 endDate, 变更后 endDate)
```

- 若日期发生变更，取前后范围的并集，确保前端旧视图和新视图都能刷新
- 全天待办：`startDate === endDate`
- 无具体日期：使用创建日期或后台约定的 fallback

### 4.6 snapshot 策略

| 阶段 | 策略 |
| --- | --- |
| P0 | 不携带 snapshot，前端收到事件后自行 GET 单条 |
| P2 | 携带 snapshot，结构与 `month-list` 返回的单项一致，减少前端 GET |

snapshot 应包含前端 normalize 所需的全部字段（status、receiveStatus、assigneeId、type 等）。

## 5. WebSocket 推送改造

### 5.1 现有通道复用

继续使用：

```
ws(s)://{host}/websocket/{userId}
```

无需新建连接；同一连接可推送多种 `type`。

### 5.2 消息类型

| type | 用途 | 优先级 |
| --- | --- | --- |
| `sys_message` | 通知文案、inbox 展示 | 保留 |
| `todo_changed` | 数据同步 | 新增 |

### 5.3 推送时机

```
DB 事务 commit 成功
  → 构建 TodoChangedEvent
  → 异步推送（不阻塞 HTTP 响应）
  → 推送失败写入重试队列
```

**禁止**在事务提交前推送（避免前端拉到未提交数据）。

### 5.4 推送规则

| 规则 | 说明 |
| --- | --- |
| 定向推送 | 按 `affectedUserIds` 逐个推送 |
| 幂等 | 同一 `eventId` 可安全重复推送 |
| 顺序 | 同一 `todoId` 的事件按 `updatedAt` 有序（尽力保证） |
| 离线 | 不保证离线期间逐条补推，由 changes 接口补偿 |

### 5.5 与 sys_message 的关系

建议**并行发送**，职责分离：

```
写操作成功
  ├── 生成 sys_message（用户可读的通知文案）
  └── 生成 todo_changed（机器可读的数据变更）
```

| 场景 | sys_message | todo_changed |
| --- | --- | --- |
| 静默后台同步（如同步日历） | 可选 | 必须 |
| 用户操作（接受/拒绝） | 必须 | 必须 |
| 仅修改非关键字段 | 可选 | 必须 |

### 5.6 推送示例

**待办创建**：

```json
{
  "type": "todo_changed",
  "eventId": "evt_20260722_a1b2c3",
  "action": "created",
  "entityType": "todo",
  "todoId": "90001",
  "mainTodoId": "90001",
  "affectedUserIds": ["10001", "10002"],
  "dateRange": { "startDate": "2026-07-22", "endDate": "2026-07-22" },
  "version": 1,
  "updatedAt": "2026-07-22T15:40:00+08:00",
  "canOpenDetail": true
}
```

**撤销待办**：

```json
{
  "type": "todo_changed",
  "eventId": "evt_20260722_d4e5f6",
  "action": "revoked",
  "entityType": "todo",
  "todoId": "90001",
  "affectedUserIds": ["10001", "10002", "10003"],
  "dateRange": { "startDate": "2026-07-20", "endDate": "2026-07-25" },
  "version": 8,
  "updatedAt": "2026-07-22T16:00:00+08:00",
  "canOpenDetail": false
}
```

## 6. 新增接口：changes 增量补偿

### 6.1 接口定义

```
GET /smart-todo/changes
```

### 6.2 请求参数

| 参数 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `since` | string | 是 | ISO 8601，上次同步时间 |
| `startDate` | string | 否 | 限定日期范围 YYYY-MM-DD |
| `endDate` | string | 否 | 限定日期范围 YYYY-MM-DD |

### 6.3 响应

```json
{
  "code": 0,
  "success": true,
  "message": "ok",
  "data": {
    "serverTime": "2026-07-22T15:45:00+08:00",
    "changed": [
      {
        "todoId": "123",
        "action": "updated",
        "entityType": "meeting",
        "mainTodoId": "123",
        "version": 7,
        "updatedAt": "2026-07-22T15:40:00+08:00"
      }
    ],
    "deleted": ["456", "789"]
  }
}
```

### 6.4 业务规则

| 规则 | 说明 |
| --- | --- |
| 权限 | 只返回当前登录用户有权查看的变更 |
| since 边界 | 返回 `updatedAt > since` 的记录（不含等于） |
| deleted | 软删除/撤销的 ID 列表 |
| 上限 | 建议单次最多 500 条，超出返回 `hasMore: true` + `nextSince` |
| 去重 | 同一 todoId 多次变更只返回最新一条 |

### 6.5 实现建议

**方案 A（推荐）**：独立变更日志表

```sql
CREATE TABLE smart_todo_change_log (
  id           BIGINT PRIMARY KEY AUTO_INCREMENT,
  event_id     VARCHAR(64) NOT NULL UNIQUE,
  todo_id      VARCHAR(64) NOT NULL,
  action       VARCHAR(32) NOT NULL,
  entity_type  VARCHAR(16) NOT NULL,
  version      INT,
  updated_at   DATETIME(3) NOT NULL,
  affected_users JSON NOT NULL,
  date_range   JSON NOT NULL,
  is_deleted   TINYINT(1) DEFAULT 0,
  created_at   DATETIME(3) DEFAULT CURRENT_TIMESTAMP(3),
  INDEX idx_user_time (updated_at),
  INDEX idx_todo (todo_id)
);
```

查询时 JOIN 用户权限过滤。

**方案 B（轻量）**：直接查主表 `update_time > since`

- 优点：无需新表
- 缺点：deleted 记录难以追踪，性能随数据量下降

## 7. 现有写接口改造清单

每个写接口在**事务提交成功后**调用 `TodoChangePublisher.publish(event)`。

### 7.1 `POST /smart-todo/create`

| 项 | 内容 |
| --- | --- |
| action | `created` |
| affectedUserIds | creator + 所有 assignee |
| snapshot | P2 起携带 |

### 7.2 `PUT /smart-todo`

| 项 | 内容 |
| --- | --- |
| action | `updated`；若接受人变更则额外发 `assignee_added` / `assignee_removed` |
| version 校验 | 请求携带 `expectedVersion`，不匹配返回 409 |
| dateRange | 取变更前后并集 |

### 7.3 `DELETE /smart-todo/{id}`

| 项 | 内容 |
| --- | --- |
| action | `deleted` |
| canOpenDetail | false |
| deleted | 加入 changes 的 deleted 列表 |

### 7.4 `POST /smart-todo/complete/{id}`

| 项 | 内容 |
| --- | --- |
| action | `status_changed` |
| affectedUserIds | creator + handler + 所有协作者 |

### 7.5 `POST /smart-todo/cancel`

| 项 | 内容 |
| --- | --- |
| action | `revoked` |
| affectedUserIds | creator + 所有有效 assignee |
| canOpenDetail | false |

### 7.6 `POST /smart-todo/accept`

| 项 | 内容 |
| --- | --- |
| action | `status_changed` |
| affectedUserIds | creator + acceptor |

### 7.7 `POST /smart-todo/reject`

| 项 | 内容 |
| --- | --- |
| action | `status_changed` |
| 备注 | sys_message 应携带拒绝原因 |

### 7.8 `POST /smart-todo/transfer`

| 项 | 内容 |
| --- | --- |
| action | `status_changed` + `assignee_removed` + `assignee_added` |
| affectedUserIds | creator + 原/新 assignee |

### 7.9 `POST /smart-todo/process`

| 项 | 内容 |
| --- | --- |
| action | `process_added` |
| 备注 | 派发人应收到通知 |

### 7.10 `POST /smart-todo/sync-calendar`

| 项 | 内容 |
| --- | --- |
| action | `created` 或 `updated`（按是否已存在） |
| entityType | `meeting` |

## 8. 版本冲突控制

### 8.1 数据模型

主待办/子待办表增加：

```sql
version INT NOT NULL DEFAULT 1
```

每次写操作 `version = version + 1`，同时更新 `update_time`。

### 8.2 更新请求

```json
PUT /smart-todo
{
  "id": "12345",
  "expectedVersion": 6,
  "title": "新标题",
  ...
}
```

### 8.3 冲突响应

```json
HTTP 409 Conflict
{
  "code": 40901,
  "success": false,
  "message": "数据已被他人修改，请刷新后重试",
  "data": {
    "currentVersion": 7,
    "updatedAt": "2026-07-22T15:40:00+08:00",
    "updatedBy": "10002"
  }
}
```

### 8.4 列表/详情返回

`month-list` 和 `GET /smart-todo/{id}` 响应增加：

```json
{
  "version": 7,
  "updateTime": "2026-07-22T15:40:00+08:00"
}
```

## 9. 列表接口增强（可选 P4）

### 9.1 dataVersion

`GET /smart-todo/month-list` 响应包装：

```json
{
  "dataVersion": "v20260722-154500",
  "items": [ ... ]
}
```

`dataVersion` = 当前用户该范围内所有待办的最大 `update_time` 哈希或时间戳。

### 9.2 ETag 支持

```
GET /smart-todo/month-list?startDate=2026-07-01&endDate=2026-07-31
If-None-Match: "v20260722-154500"

→ 304 Not Modified（无变更）
→ 200 + 新 ETag（有变更）
```

前端轻量对账时使用，减少全量 payload。

## 10. 幂等与去重

### 10.1 写操作幂等

建议支持请求头：

```
Idempotency-Key: {uuid}
```

同一 key 重复提交返回相同结果，不重复创建待办/消息/事件。

### 10.2 事件去重

- `eventId` 全局唯一（DB UNIQUE 约束）
- 重复 publish 同一 eventId 应 no-op

### 10.3 消息去重

sys_message 生成时携带业务去重键：

```
dedupeKey = {bizType}:{bizId}:{action}:{targetUserId}
```

避免同一操作产生多条相同通知。

## 14. 后台自测清单

| # | 场景 | 预期 |
| --- | --- | --- |
| 1 | create 待办 | creator + assignee 均收到 todo_changed |
| 2 | A 完成，B 在线 | B 收到 status_changed |
| 3 | 修改接受人 A→B | A 收到 assignee_removed，B 收到 assignee_added |
| 4 | 撤销协作待办 | 所有有效 assignee 收到 revoked，canOpenDetail=false |
| 5 | sync-calendar | 相关用户收到 created/updated，entityType=meeting |
| 6 | 断线 5min 后 changes | since 之后变更全部返回 |
| 7 | 重复 eventId publish | 不报错，不产生副作用 |
| 8 | 并发 PUT version=6/7 | 后提交者收到 409 |
| 9 | 无权限用户 changes | 不包含无权待办 |
| 10 | 删除待办 | changes.deleted 包含该 ID |

## 16. 监控与运维

| 指标 | 说明 |
| --- | --- |
| ws_push_total | 推送总数 |
| ws_push_failed | 推送失败数 |
