# o4.8 · 项目代码结构优化方案

> 目标：在**完全不改变现有 UI 效果与交互逻辑**的前提下，把代码结构改造得更健壮、可持续、易维护。
> 原则：只做结构性重构（搬移、抽取、立规矩），不动业务行为。

---

## 0. 执行总原则（重构纪律）

1. **不破坏 UI / 交互**：只重组代码结构，不改变任何渲染结果和交互行为。
2. **小步快跑**：每一步只做一类改动，改动范围可控、可回滚。
3. **每步先验证再提交**：每步都跑 `pnpm typecheck` + `pnpm test`，全绿后再 `git commit`，绝不留半成品。
4. **进度可续**：每个安全步骤单独提交，中途中断也不丢进度，重开后从最后一个 commit 继续。
5. **先立规矩，后拆大件**：先把"模块边界 / 依赖约束"立起来，避免拆完又乱。

> 关于"关机也不停"：本地 agent 无法在关机/休眠的机器上运行。真正的无人值守需用 Cursor 云端/后台 Agent（远程 VM 运行）。本方案采用"每步提交"来对抗中断风险。

## 2. 问题清单

### P0 · 模块边界失控（最严重，会越滚越乱）

1. **分层放错——登录依赖 home 的待办服务**
   `auth/LoginPage.vue`、`auth/DesktopAuthConfirmDialog.vue` 从 `home/dashboard/todo.service` 导入 `loginSmartTodo` / `loadCurrentUser`。
   登录、取当前用户属于**更底层能力**，却埋在 home 里，依赖方向反了。

2. **循环依赖**
   - `home/dashboard/dashboardTools.ts` → `agent-center/links`
   - `agent-center/index.vue` → `home/dashboard/DashboardTopBar.vue`
   home ⇄ agent-center 互相依赖。

3. **共享组件放在业务模块里**
   `DashboardTopBar.vue` 被 `home`、`agent-center` 共用，本质是"应用外壳（layout）"，不该属于 home。

4. **无模块公共 API**
   所有跨模块 import 都深钻内部文件（`@/modules/home/dashboard/xxx`），任何模块都能引用任何模块的任意内部文件——失控根源。

### P1 · CSS 无法复用

- `backdrop-filter`（玻璃拟态）散落在 **14 个文件**：NotificationCenter 13 处、DetailedDashboard 12 处、TopBar 8 处、CalendarWorkspace 6 处……全是 scoped 各写各的，无统一变量/公共类。
- 样式技术栈"多头"：同时装了 `tailwindcss` + `daisyui` + `@inspira-ui/plugins` + `tw-animate-css` + `sass` + 手写 CSS 变量，心智负担重。

### P1 · 上帝组件

- `DayPreviewPanel.vue`（3110 行）一个组件承担 **查看/创建/编辑/删除/AI 解析** 五种模式。
- `CalendarWorkspace.vue`（总览）与 `DetailedDashboardWorkspace.vue`（详细）是两份平行实现，**待办 CRUD / 接受 / 拒绝 / 详情加载逻辑几乎复制了两份**（相同的 state、相同的 handler、相同的 service import、相同的 `DayPreviewPanelExpose` 类型）。改一个 bug 要改两处。

### P2 · 其他

- **业务数据硬编码**：2026 节假日 / 调休 / 节气写死在 `CalendarWorkspace.vue`，跨年失效。
- **死代码**：`home/dashboard/components/CalendarWeekTimeline.vue`（378 行，确认无任何引用）。
- **一次性产物**：根目录 `fix_padding.py`、`replace_bubble.py`、`rewrite_dashboard.py`、`tode.txt`、`verification-dashboard.png`。
- **依赖可能冗余**：`daisyui`、`@inspira-ui/plugins`、`gsap`、`motion-v`、`tw-animate-css`、`sass` 等需审计是否仍在用。

### 值得肯定
数据层分得不错：`todo.service.ts`、`todoDisplay.ts`、`*.helpers.ts` 已从组件独立且有测试覆盖。屎山主要集中在 **.vue 视图层**（CSS + 双份模式逻辑）。

## 3. 优化方案

### 3.1 CSS 三层化（回答"要不要抽成单独文件夹"）

**结论：要抽，但按"复用范围"分三层，不是一锅端全局。** 目标是消除"重复"，不是消除"就近"。

**第 1 层 · 全局设计令牌（`styles/`，强化现有）**
把玻璃拟态三件套固化为 CSS 变量：

```css
/* styles/variables.css 补充 */
--glass-blur: 18px;
--glass-saturate: 140%;
--glass-border: rgba(255,255,255,.18);
```

**第 2 层 · 可复用"视觉原语"类（新增文件夹）**

```
src/styles/
  base/        reset.css, theme.css, transitions.css
  tokens/      variables.css（设计令牌）
  components/  glass.css, card.css, scrollbar.css   ← 新增：跨组件复用类
  index.css    （@import 汇总）
```

```css
/* styles/components/glass.css */
@layer components {
  .glass-panel {
    background: var(--app-panel-bg);
    backdrop-filter: blur(var(--glass-blur)) saturate(var(--glass-saturate));
    border: 1px solid var(--glass-border);
    box-shadow: var(--app-shadow-soft);
  }
}
```

各组件删掉重复的 `backdrop-filter` 块，改用 `class="glass-panel"`——**一处改，全站生效**。

**第 3 层 · 组件私有样式（留在 `.vue` 的 `<style scoped>`）**
只跟该组件相关、不被别人复用的样式就近保留。若某组件 scoped CSS 过大（如 DayPreviewPanel 1800 行），解决办法是**拆子组件**（每个子组件带自己那份样式），而不是把 CSS 搬到全局文件夹。

### 3.2 模块内部统一结构 + 公共出口

给每个 module 统一内部结构，并用 `index.ts` 作为唯一对外出口：

```
modules/<feature>/
  index.ts            # ← 模块唯一公共出口（barrel），外部只能 import 这里
  routes.ts
  pages/              # 路由级页面
  components/         # 模块私有组件
  composables/        # use* 组合逻辑
  services/           # 接口层
  stores/             # 模块内 store（非全局）
  helpers/ types.ts   # 纯函数 + 类型
```

两条硬约束（防止继续烂下去，**最关键**）：
1. **模块只能通过 `index.ts` 暴露**，用 ESLint `no-restricted-imports` 禁止 `@/modules/x/内部深路径`。
2. **加语义化别名**：`@home`、`@shared`、`@ui`、`@styles`，替代满屏 `../../`。

### 3.3 具体调整对照表

| 现状问题 | 建议动作 | 收益 |
|---|---|---|
| 登录/用户接口埋在 `home/todo.service` | 抽出 `auth/services/auth.service`（或 `shared`）：登录、getInfo、当前用户 | 打破 auth→home 错误依赖 |
| `DashboardTopBar` 被多模块共用 | 提升为 `src/layouts/AppShell` 或 `shared/components/layout/` | 消除 home⇄agent-center 循环 |
| `dashboardTools`→`agent-center/links` | 跳转配置下沉到 `shared/config` 或后端下发 | 断开循环依赖 |
| 三个 2500+ 行组件 | 拆子组件 + 抽 composable | 可读、可测 |
| 重复 CSS | 抽第 2 层公共类 | 一处改全站生效 |
| 硬编码节假日 / mock | 数据外置 / 走接口 | 跨年不失效 |
| 死代码、一次性脚本 | 删除 | 减负 |
| 依赖冗余 | 审计未使用依赖 | 构建更轻 |

### 3.4 巨型组件拆分与逻辑抽取

- **抽 composable（收益最大）**：把总览/详细两份重复逻辑抽成共享 composable：
  - `useTodoPanelActions()`：接受/拒绝/完成/删除/详情加载 + 相关 processing 状态。
  - `useDayPreviewController()`：面板 view/edit/create 模式切换、dirty 状态、丢弃确认。
  - `useCalendarNavigation()`：`now/currentMonth/calendarViewMode` + 月周切换 + 加载区间计算。
  - 预计各消除 300~500 行重复，bug 只需修一处，且 composable 可单测。
- **拆 DayPreviewPanel**：拆成 `DayPreviewView.vue` / `DayPreviewForm.vue`（编辑创建共用）/ `DayPreviewAiParse.vue`，主文件只做壳与模式路由，目标 < 800 行。

## 4. 落地路线图（低风险 → 高价值）

| 阶段 | 内容 | 风险 | 验证方式 |
|---|---|---|---|
| **① 零风险清理** | 删死代码 `CalendarWeekTimeline.vue`、根目录一次性脚本/产物 | 极低 | typecheck + test + 构建 |
| **② 立规矩** | 加语义化别名；各模块加 `index.ts` 出口；ESLint 禁深 import | 低（增量） | typecheck + lint |
| **③ 修分层** | 抽 `auth.service`；`DashboardTopBar` 提为 layout；断循环依赖 | 中 | typecheck + test + 手测登录/顶栏 |
| **④ CSS 三层化** | 建 `styles/components/`，抽 `glass-panel` 等公共类，逐组件替换 | 中（视觉需逐个核对） | 逐组件视觉回归对比 |
| **⑤ 拆巨型组件** | 抽 `useTodoPanelActions` 等 composable；拆 DayPreviewPanel | 较高 | test + 全量手测交互 |
| **⑥ 数据外置** | 节假日/节气抽独立数据文件或走接口 | 低 | 日历显示核对 |

> 建议执行顺序即①→⑥。**优先级最高的不是拆大文件，而是先立规矩（②）**，否则拆完还会再乱。

## 5. 每步验证清单（每个 commit 前必过）

- [ ] `pnpm typecheck` 通过（`vue-tsc --noEmit`）
- [ ] `pnpm test` 全绿（vitest）
- [ ] `pnpm build` 可构建（涉及别名/路径改动时）
- [ ] 涉及视觉/交互的改动：人工对比改动前后页面无差异
- [ ] 该步骤单独 `git commit`，信息说明"结构重构，不改行为"

---

## 6. 风险与回滚

- 每步独立提交，任何一步出问题可 `git revert` 单个 commit，不影响其他进度。
- CSS 替换（阶段④）与组件拆分（阶段⑤）风险最高，采用"逐个组件替换 + 逐个核对"策略，不批量一次性替换。
- 全程不改动 `template` 的结构语义与 `script` 的业务逻辑，仅做搬移与抽取。
