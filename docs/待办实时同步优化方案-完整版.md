## 1. 背景与问题

### 1.1 现状

当前系统在多人协作或多端同时使用时，存在**待办/会议信息不能及时更新**的问题，用户需要手动刷新页面才能看到最新数据。

### 1.2 根因分析

| 层级 | 现状 | 问题 |
| --- | --- | --- |
| 数据加载 | `dashboard-todos.store` 按日期范围加载并缓存 | 同范围内默认不重新请求 |
| WebSocket | 已有 `/websocket/{userId}`，推送 `sys_message` | 只刷新站内消息和待接受列表，**不刷新日历列表** |
| 日历刷新 | `reloadCurrentRange({ force: true })` | 仅在手动操作、接受/拒绝待办时触发 |
| 详情面板 | `useTodoDetailCache` 有缓存 | 列表更新后详情可能仍是旧数据 |
| H5 | `useH5Todos` 仅初始化 + 手动刷新 | 无自动同步机制 |
| 连接管理 | WebSocket 挂在通知中心组件内 | 与 UI 耦合，生命周期不够稳定 |

### 1.3 目标

建立 **「变更 → 事件 → 推送 → 增量更新 UI」** 的完整链路：

- 多人同时在线时，一方操作后另一方**无需手动刷新**即可看到结果
- WebSocket 断线重连后，数据**最终与后台一致**
- 同一事件重复推送**不产生重复列表项或重复通知**
- 轮询仅作为断线兜底，不作为主要同步手段

## 3. 后台方案摘要

> 完整细节见 [待办实时同步优化方案-后台.md](./待办实时同步优化方案-后台.md)

### 3.1 核心改动

1. **统一变更事件**：所有待办/会议写操作成功后发布 `todo_changed` 事件
2. **扩展 WebSocket 协议**：在现有 `sys_message` 之外新增 `todo_changed` 类型
3. **新增增量补偿接口**：`GET /smart-todo/changes?since=...`
4. **版本冲突控制**：详情/更新接口增加 `version`，冲突返回 `409`
5. **列表接口增强（可选）**：`month-list` 返回 `dataVersion` 或支持 ETag

### 3.2 涉及现有写接口

| 接口 | 动作 |
| --- | --- |
| `POST /smart-todo/create` | 创建 |
| `PUT /smart-todo` | 更新 |
| `DELETE /smart-todo/{id}` | 删除 |
| `POST /smart-todo/complete/{id}` | 完成 |
| `POST /smart-todo/cancel` | 取消/撤销 |
| `POST /smart-todo/accept` | 接受 |
| `POST /smart-todo/reject` | 拒绝 |
| `POST /smart-todo/transfer` | 转交 |
| `POST /smart-todo/process` | 进展更新 |
| `POST /smart-todo/sync-calendar` | 会议同步 |

### 3.3 实施优先级

| 阶段 | 内容 | 优先级 |
| --- | --- | --- |
| P0 | 所有写操作发 `todo_changed` WebSocket | 最高 |
| P1 | `GET /smart-todo/changes` 补偿接口 | 高 |
| P2 | `snapshot` 字段减少前端 GET | 中 |
| P3 | `version` + 409 冲突检测 | 中 |
| P4 | `month-list` ETag / dataVersion | 低 |

## 4. 前端方案摘要

> 完整细节见 [待办实时同步优化方案-前端.md](./待办实时同步优化方案-前端.md)

### 4.1 核心改动

1. **新建 `dashboard-sync.service.ts`**：统一管理 WebSocket、事件去重、断线补偿
2. **新建 `useDashboardDataSync.ts`**：页面级挂载，驱动 store / 详情 / inbox 联动
3. **Store 增量能力**：`upsertEvent` / `patchEvent` / `removeEvent` / `reloadEvent`
4. **WebSocket 上提**：从 `useDashboardNotifications` 剥离，进首页即建连
5. **详情冲突处理**：编辑中收到远端更新时提示冲突
6. **H5 复用**：PC 与 H5 共用 sync 服务
7. **多 Tab 协调（可选）**：`BroadcastChannel` 避免重复连接

### 4.2 新增/修改文件

| 文件 | 操作 |
| --- | --- |
| `services/dashboard-sync.service.ts` | 新增 |
| `composables/useDashboardDataSync.ts` | 新增 |
| `stores/dashboard-todos.store.ts` | 增量 patch + silent 刷新 |
| `composables/useTodoDetailCache.ts` | 版本冲突 |
| `composables/useDashboardNotifications.ts` | 剥离 WS，只消费 inbox |
| `DashboardPage.vue` | 挂载 sync |
| `CalendarWorkspace.vue` / `DetailedDashboardWorkspace.vue` | 详情跟随 |
| `h5/composables/useH5DataSync.ts` | 新增 |

## 5. 事件协议（前后端共识）

### 5.1 WebSocket：`todo_changed`

```json
{
  "type": "todo_changed",
  "eventId": "evt_8f3a2b1c",
  "action": "created",
  "entityType": "todo",
  "todoId": "12345",
  "mainTodoId": "12345",
  "childTodoId": null,
  "affectedUserIds": ["10001", "10002"],
  "dateRange": {
    "startDate": "2026-07-01",
    "endDate": "2026-07-31"
  },
  "version": 7,
  "updatedAt": "2026-07-22T15:40:00+08:00",
  "canOpenDetail": true,
  "snapshot": {}
}
```

### 5.2 字段说明

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| `type` | 是 | 固定 `todo_changed` |
| `eventId` | 是 | 全局唯一，客户端去重用 |
| `action` | 是 | 见下方 action 枚举 |
| `entityType` | 是 | `todo` / `meeting` |
| `todoId` | 是 | 前端列表/详情主键 |
| `mainTodoId` | 否 | 主待办 ID（子待办场景） |
| `childTodoId` | 否 | 子待办 ID |
| `affectedUserIds` | 是 | 需收到推送的用户工号列表 |
| `dateRange` | 是 | 受影响日期范围，前端判断是否刷新当前视图 |
| `version` | 推荐 | 并发控制 |
| `updatedAt` | 是 | ISO 8601 |
| `canOpenDetail` | 推荐 | 撤销/删除后消息是否可跳转详情 |
| `snapshot` | 否 | 变更后实体快照，有则前端可直接 patch |

### 5.3 action 枚举

| action | 触发场景 |
| --- | --- |
| `created` | 新建待办/会议 |
| `updated` | 编辑标题、时间、内容、接受人等 |
| `deleted` | 删除 |
| `status_changed` | 接受、拒绝、完成、取消等状态变更 |
| `assignee_added` | 新增接受人 |
| `assignee_removed` | 移除接受人 |
| `revoked` | 派发人撤销 |
| `process_added` | 新增进展 |

> 与 [待办协作体验优化需求清单](./待办协作体验优化需求清单.md) TODO-006 建议事件类型对齐。

### 5.4 保留现有 `sys_message`

`sys_message` 继续负责**通知文案和 inbox 展示**；`todo_changed` 负责**数据同步**。两者可同时推送，职责分离：

- `sys_message` → 消息中心、角标、Toast
- `todo_changed` → 日历列表、详情、待接受列表

## 6. 增量补偿接口

### 6.1 请求

```
GET /smart-todo/changes?since=2026-07-22T15:00:00+08:00&startDate=2026-07-01&endDate=2026-07-31
```

### 6.2 响应

```json
{
  "code": 0,
  "success": true,
  "data": {
    "serverTime": "2026-07-22T15:45:00+08:00",
    "changed": [
      {
        "todoId": "123",
        "action": "updated",
        "entityType": "meeting",
        "version": 7,
        "updatedAt": "2026-07-22T15:40:00+08:00"
      }
    ],
    "deleted": ["456"]
  }
}
```

### 6.3 使用场景

- WebSocket 断线期间的事件补偿
- 重连成功后 `since=lastSyncedAt` 对账
- 页面从后台切回且 WS 不可用时兜底

## 7. 端到端数据流

### 7.1 正常路径（WebSocket 在线）

```
用户 A 编辑待办
  → PUT /smart-todo 成功
  → 后台发布 todo_changed
  → WebSocket 推送给 affectedUserIds 中的 B、C
  → B 的前端 sync 服务收到事件
  → 判断 dateRange 与 loadedRange 重叠
  → store.patchEvent / upsertEvent（有 snapshot）
    或 store.reloadEvent（无 snapshot，单条 GET）
  → 日历 UI 即时更新
  → 若 B 正打开该详情 → 静默刷新详情
  → 并行：sys_message 更新 inbox 和角标
```

### 7.2 断线路径

```
WebSocket onclose
  → 启动 changes 轮询（30s，仅 visible 时）
  → GET /smart-todo/changes?since=lastSyncedAt
  → 逐条 apply 到 store
  → WebSocket 重连成功
  → 立即再拉一次 changes 对账
  → 停止轮询
```

### 7.3 并发编辑路径

```
用户 B 正在编辑 todoId=123（本地 version=6）
  → 收到 todo_changed，version=7
  → 检测到编辑中 + 远端更新
  → 提示「数据已被他人修改，请刷新后重试」
  → 禁用保存，提供「重新加载」按钮
```

## 8. 联调场景

### 场景 A：双人在线实时同步

1. 用户 A、B 同时打开首页日历
2. A 创建待办并指派给 B
3. B 无需刷新，日历和待接受列表出现新待办
4. B 接受后，A 的日历状态同步更新

### 场景 B：会议同步

1. A 通过 `sync-calendar` 同步会议
2. B 日历中对应日期出现/更新会议条目
3. inbox 收到 `bizType=2` 的 sys_message

### 场景 C：断线恢复

1. B 断网 5 分钟
2. 期间 A 完成、编辑多条待办
3. B 恢复网络
4. changes 接口补拉遗漏事件，数据与 A 一致

### 场景 D：撤销与详情失效

1. A 撤销协作待办
2. B 若正在查看详情 → 关闭操作区，提示「待办已被撤销」
3. sys_message 中 `canOpenDetail=false`，点击显示快照

### 场景 E：重复推送幂等

1. 同一 `eventId` 推送两次
2. 前端只处理一次，列表无重复项

### 场景 F：H5 与 PC 互通

1. A 在 PC 编辑，B 在 H5 查看
2. B 的 H5 页面同步更新

## 9. 验收标准

### 9.1 功能

- [ ] 所有写操作均产生 `todo_changed` 推送
- [ ] 两个用户同时在线，操作后 3 秒内对方可见（网络正常）
- [ ] 断线重连后数据最终一致
- [ ] 同一 eventId 重复推送不重复处理
- [ ] 撤销/删除后详情不可继续操作
- [ ] 简约模式、详细模式、H5 结果一致

### 9.2 性能

- [ ] WebSocket 在线时不周期性全量 `month-list`
- [ ] 有 snapshot 时不额外 GET 详情
- [ ] changes 接口响应 < 500ms（100 条变更内）

### 9.3 体验

- [ ] 后台同步不触发 blocking loading
- [ ] 编辑冲突有明确提示
- [ ] 页面切回前台时静默对账

## 10. 实施计划

| 阶段 | 周期 | 后台 | 前端 | 产出 |
| --- | --- | --- | --- | --- |
| 一期 P0 | 1 周 | `todo_changed` 推送（核心写操作） | sync 服务 + store patch + 页面挂载 | 主路径实时同步 |
| 二期 P1 | 1 周 | changes 接口 | 断线补偿 + 重连对账 | 断线最终一致 |
| 三期 P2-P3 | 1 周 | snapshot + version/409 | 详情冲突 + 编辑保护 | 多人编辑安全 |
| 四期 P4 | 可选 | ETag | 轻量对账 | 降低补偿成本 |

## 11. 风险与依赖

| 风险 | 缓解 |
| --- | --- |
| 后台短期无法改接口 | 先用 [纯前端补救方案](./待办协作体验优化需求清单.md)（WebSocket bizType 触发刷新 + 智能轮询） |
| 推送遗漏 | changes 接口 + 重连对账 |
| 多 Tab 重复连接 | BroadcastChannel 协调 |
| 主/子待办 ID 混乱 | 事件携带 mainTodoId + childTodoId，前端统一映射 |
| 会议与待办规则不同 | entityType 区分，后台明确哪些 action 适用会议 |
