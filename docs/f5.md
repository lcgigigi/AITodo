# f5 — 前端架构优化方案总结

更新时间：2026-07-08
完整版详见 [architecture-optimization-plan.md](./architecture-optimization-plan.md)

---

## 1. 总体判断

项目不算真正的"屎山"：模块划分、HTTP 层、vitest 测试、全局 token 底子都在。
真正的问题是**缺少边界约束**——新功能不断堆进 home 少数几个大文件，导致：

> **home 变成全项目的核心单体，几个文件太大，而不是文件太多。**

---

## 2. 四个核心问题

### 2.1 巨型单文件组件（P0）

全项目 `src/` 约 3.4 万行，Top 5：

| 文件 | 行数 |
|------|------|
| `home/dashboard/DetailedDashboardWorkspace.vue` | 3183 |
| `home/dashboard/DayPreviewPanel.vue` | 3110 |
| `home/dashboard/CalendarWorkspace.vue` | 2584 |
| `leader-board/index.vue` | 1889 |
| `home/dashboard/CalendarMonth.vue` | 1534 |

一个文件同时干：数据加载 + 筛选 + CRUD + 弹层 + 快捷键 + 上千行 CSS。

**规矩：单文件理想 < 500 行，超过 800 行必须拆分（建议 CI 检查）。**

### 2.2 home 单体，模块边界被打穿（P0）

实际存在的跨模块"偷 import"：

```
auth/LoginPage.vue              → home/dashboard/todo.service.ts
auth/DesktopAuthConfirmDialog   → home/dashboard/todo.service.ts
agent-center/index.vue          → home/dashboard/DashboardTopBar.vue
stores/dashboard-todos.store    → home/dashboard/todo.service.ts
suggestion-inbox                → home/dashboard/suggestion.service.ts
```

后果：登录页无法脱离 home；改顶栏会波及智能体中心；模块划分名存实亡。

### 2.3 重复业务逻辑（P0）

- 两个 workspace（总览/详细）约 **40–50% 逻辑复制粘贴**：待办接受/拒绝/删除/切换、详情缓存、AI 解析创建流程
- 待办筛选逻辑 **3 套实现**（`dayPreviewPanel.helpers.ts` / `CalendarWorkspace` / `DetailedDashboardWorkspace`）
- 日历 **2 套实现**（总览内联周条+月历 vs `CalendarMonth.vue`）

### 2.4 样式体系混乱（P1）

- `.vue` 内 `<style>` 块总计约 **14,479 行，占源码 42%**
- 技术栈选了 Tailwind + shadcn，页面主力却是传统巨型 scoped CSS
- `--app-primary` / `--todo-primary` 等 token 重复定义
- DaisyUI 已安装但**源码零引用**，纯配置噪音

### 2.5 其他

| 问题 | 说明 |
|------|------|
| 死代码 | `CalendarWeekTimeline.vue`（378 行，零引用） |
| 硬编码 | 2026 节假日/节气写死在 `CalendarWorkspace.vue` |
| 全能文件 | `todo.service.ts` 1086 行：认证 + DTO 归一化 + CRUD + AI 解析焊在一起 |
| ECharts 重复 | `agent-center` 与 `leader-board` 各写一套 init/resize |
| Toast 不统一 | 一处包装 `showToast`，一处直接用 `feedbackStore` |

## 3. 目标架构：轻量 Feature-Sliced

```
src/
├── app/          # 应用启动层（保持）
├── pages/        # 路由薄页面，只做组装，<200 行
├── widgets/      # 跨页面组合块（AppTopBar、通知中心、新手引导）
├── features/     # 可复用业务能力（todo / calendar / sys-message / suggestion / token-usage）
├── entities/     # 领域实体，纯数据+规则（user / todo / message）
├── shared/       # api、ui、composables、lib、constants
├── styles/       # 三层样式体系（见第 4 节）
├── stores/       # 仅真正全局的 store
└── router/
```

**依赖硬规则（写进 ESLint）：**

1. `pages → widgets/features → entities → shared`，只能向下依赖
2. `features` 之间不能互相 import
3. **auth 不能依赖 home/todo 的任何文件**
4. 单文件 >500 行 warn，>800 行 fail

`modules/` 逐步退役，渐进迁移，不推倒重来。

---

## 4. 样式方案：抽离但不搬家

**结论：CSS 要抽，但不是抽成一个 `css/` 大杂烩，而是三层分工：**

| 层级 | 位置 | 内容 | 原则 |
|------|------|------|------|
| L1 foundation | `styles/foundation/` | reset、variables、theme | 全局 token，已有可延续 |
| L2 patterns | `styles/patterns/`（Tailwind `@layer components`） | 玻璃面板、卡片壳、筛选条、表单区 | **复用 ≥2 次才提升** |
| L3 scoped | `.vue` 内 | 仅组件私有微调 | 控制在 50–100 行 |

建议新增 patterns：`glass-surface.css`、`panel-card.css`、`filter-bar.css`、`form-module.css`、`page-shell.css`。

Token 统一到 `variables.css`（`--todo-primary: var(--app-primary)` 等）。

**新代码约定：Tailwind 布局 + patterns 语义类 + 少量 scoped。旧代码不动，触达时再迁。**

## 5. 关键拆分点

### todo.service.ts → entities/todo/

```
entities/todo/
  api/       client.ts / auth.api.ts / todo.api.ts / todo-ai.api.ts
  model/     types.ts / backend.types.ts / normalize.ts / date-range.ts
  lib/       status.ts
  index.ts   # 统一导出，兼容旧 import
```

### 两个 workspace 共用 composable

| Composable | 用途 |
|------------|------|
| `useTodoTaskActions` | accept / reject / delete / toggle |
| `useTodoDetailCache` | 详情缓存、loadDetail |
| `useTodoFilters` | 三套筛选合一 |
| `useDayPreviewHost` | 单日面板宿主逻辑 |
| `useECharts` | 图表 init / resize / dispose 统一 |

### DayPreviewPanel 拆分（最大 ROI，3110 → ~400 行）

```
features/todo/ui/day-preview/
  DayPreviewPanel.vue          # 薄壳
  DayPreviewList.vue           # 列表 + 筛选
  DayPreviewForm.vue           # 创建/编辑/查看
  DayPreviewFormSchedule.vue   # 时间/全天/截止模式
```

### 日历统一

总览模式改用 `CalendarMonth` + 抽出的 `HomeWeekStrip`，消灭双实现；节假日数据抽到 `calendar-special-days/2026.ts`。

## 8. 一句话总结

**先立规矩（边界 + 行数护栏），再抽公共能力打破 home 单体，最后拆大文件，全程渐进式，不推倒重来。**

方向：pages 变薄 → features 承载业务 → widgets 放跨页 UI → entities 承载领域 → 样式三层分工。
